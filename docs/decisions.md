# Decisions

Architecture decision records — why things are the way they are.

---

## ADR-001: Interface-first design for all major components

**Decision**: Every component that a consumer might want to mock or swap exposes a Go interface (prefixed with `I`). Concrete implementations are not exported directly.

**Rationale**: Consumer services (go-dutch, go-zap) use dependency injection via interfaces. This enables unit testing with mockery-generated mocks without any HTTP/database infrastructure. It also makes the library's contracts explicit and stable.

**Tradeoff**: More boilerplate than direct struct usage. Justified by the testability benefit.

---

## ADR-002: Argon2id for password hashing

**Decision**: `tool/crypto/` uses Argon2id (via `golang.org/x/crypto/argon2`) for password hashing rather than bcrypt.

**Rationale**: Argon2id is the winner of the Password Hashing Competition and is recommended by OWASP for new systems. It is resistant to both GPU cracking and side-channel attacks. Configurable memory and CPU cost parameters allow tuning for the deployment environment.

**Tradeoff**: Slightly more complex implementation than bcrypt. The added security justifies it. Constant-time comparison prevents timing attacks.

## ADR-003: Middleware uses `http.HandlerFunc` not `http.Handler`

**Decision**: Middleware signature is `func(next http.HandlerFunc) http.HandlerFunc` rather than `func(next http.Handler) http.Handler`.

**Rationale**: `http.HandlerFunc` is a concrete function type that satisfies `http.Handler`. Using it directly avoids the boilerplate of calling `.ServeHTTP()` or wrapping lambdas in `http.HandlerFunc()`. Consumer code is cleaner.

**Tradeoff**: Slight incompatibility with third-party middleware that uses the `http.Handler` signature. A thin adapter is needed in those cases.

## ADR-004: `reqctx` for request-scoped values instead of raw context keys

**Decision**: Request-scoped values (request ID, user ID, error) are stored under a single typed key using the `reqctx` package rather than individual `context.WithValue` calls with package-level keys.

**Rationale**: Raw context keys are stringly-typed and easy to collide. A single struct in context (`*RequestContext`) groups all request-scoped values, provides typed accessors, and avoids key collision with other libraries.

**Tradeoff**: All middleware must be aware of the `reqctx` package. The `ReqCtx()` middleware must be in the chain before any middleware that uses the context values.

## ADR-005: Sensitive field redaction in logging middleware

**Decision**: The `redactor.go` middleware allows configuring JSON field paths to redact from request/response body logs.

**Rationale**: Logging request bodies is useful for debugging but risks leaking PII (passwords, tokens, personal data). Rather than disabling body logging, field-level redaction lets us log safely.

**Tradeoff**: Redaction adds CPU overhead per request. Acceptable given the small number of sensitive fields in practice.

---

## ADR-006: `chain.go` applies middleware in reverse order

**Decision**: `Chain(handler, A, B, C)` results in execution order `A → B → C → handler → C → B → A` (not `C → B → A → handler`).

**Rationale**: This is the conventional middleware chain order — the first middleware listed is the outermost wrapper and gets first/last access to the request and response. This matches the expectation of most Go developers familiar with frameworks like Chi, Gorilla, or Echo.

**Impact**: When adding middleware, the conceptually "first" middleware (e.g., request ID injection) should be listed first in `Chain()`.

---

## ADR-007: mockery for mock generation

**Decision**: Mock implementations are generated by `mockery` via `.mockery.yaml` rather than being written manually or using `gomock`.

**Rationale**: Mockery generates clean, readable mocks with no code generation annotations needed in the interface source. The `.mockery.yaml` config file makes mock generation reproducible and version-controlled.
