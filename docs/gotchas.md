# Gotchas

Traps, quirks, and things not to change without care.

---

## Breaking changes affect all consumer services

go-dutch and go-zap both depend on this library. Removing or renaming an exported symbol, changing a function signature, or changing interface methods is a **breaking change** that requires coordinated updates in both consumers. Always add new functionality without removing the old form first (introduce new constructors, don't modify existing ones).

---

## `ReqCtx()` middleware must be in the chain before other middleware that reads context

`reqctx.GetValue(ctx)` panics if the request context value is not set. The `ReqCtx()` middleware must run before any middleware that calls `reqctx.GetValue()` (e.g., logging, error handler, auth middleware). In `Chain()` calls, list `ReqCtx()` first.

---

## `Chain()` applies middleware in reverse order — not the listed order

`Chain(handler, A, B, C)` results in `A` being the outermost wrapper (first to receive the request, last to see the response), and `C` being innermost (last to receive the request). If you list auth middleware after logging middleware, the log entry may not include auth context. Think carefully about ordering.

---

## Argon2id parameters affect hash format compatibility

`tool/crypto/` encodes the Argon2id parameters (memory, iterations, parallelism, salt) into the stored hash string (PHC format). If you change the default parameters, **existing hashes remain verifiable** (the parameters are decoded from the stored string), but new hashes will use the new parameters. Never change the encoding format — it would silently break verification of all existing password hashes.

---

## `ICrypto.Verify()` always uses constant-time comparison

The `Verify` method uses a constant-time byte comparison (from `crypto/subtle`) to prevent timing attacks. Do not replace it with a simple `==` comparison for "simplicity" — it would introduce a timing oracle.

---

## Mocks in `mocks/` are auto-generated — never edit them

Files in `mocks/` are generated by `mockery`. Manual edits will be overwritten on the next `mockery --all`. If a mock needs additional behavior, use `testify/mock`'s `.On()` and `.Return()` in the test code itself.

---

## Module version must be tagged for consumers to pin

Consumer services pin a specific `v1.x.x` tag in their `go.mod`. Changes pushed to `main` without a new tag are not picked up by consumers. After a meaningful release, tag the commit:

```bash
git tag v1.x.x
git push origin v1.x.x
```

Breaking changes require a new major version tag (`v2.x.x`) and a new module path.

---

## The `main.go` at the root is an example, not a service

The `main.go` file demonstrates library usage with a sample HTTP server. It is not a production binary. Do not treat it as the library's entry point.

---

## AWS S3 integration requires a real AWS config in tests

`tool/aws/` uses the AWS SDK v2. Unit tests that exercise S3 paths need either a real AWS config (credentials) or a mock. The `IS3` interface allows mocking — always use the interface in consumer code, never the concrete type.
